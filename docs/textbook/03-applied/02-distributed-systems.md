# 第 16 章 分散システム

## 学習目標

- 「複数のマシンで協調して 1 つのシステムを作る」ときに生じる根本的な難しさを理解する。
- 時間と順序、レプリケーション、一貫性モデル、合意アルゴリズムを体系的に把握する。
- 障害モデル、冪等性、メッセージング保証、CAP / PACELC を実務で使える形で理解する。

## 16.1 なぜ分散システムか

スケール、可用性、地理的分散、コスト――現代のサービスはほぼすべて分散システム。しかし「**分散システムは難しい**」と Lamport が言ったとおり、ネットワーク分断・部分障害・時計のずれ・再順序化など、単一マシンでは出会わない現象が起きる。フォールアシーが「Fallacies of Distributed Computing」としてまとめた誤解はいまも繰り返されている。

### 分散の 8 つの誤謬

1. ネットワークは信頼できる。
2. 遅延はゼロ。
3. 帯域は無限。
4. ネットワークは安全。
5. トポロジーは変化しない。
6. 管理者は 1 人。
7. 転送コストはゼロ。
8. ネットワークは均質。

すべて誤り。これらを否定するところから設計が始まる。

## 16.2 時間と順序

物理時計は同期しない（NTP でも数 ms ずれる）。

### Lamport 時計

各イベントに整数を割当てる。送信側のイベント時刻 + 1 を受信側で取り込む。**因果律** を保つ全順序を作れるが、独立イベントの順序は本来の意味を持たない。

### ベクトル時計

各ノードが他全ノードの時計を持つベクトル。「$V_a < V_b$ ⇔ $a$ が $b$ の原因」を判定できる。

### Hybrid Logical Clock (HLC)

物理時計と論理時計のハイブリッド。CockroachDB などで採用。

### Google TrueTime

GPS と原子時計で時刻の不確実性区間を返す。Spanner の外部一貫性の鍵。

## 16.3 障害モデル

- **Crash-stop**: ノードが止まる（再起動しない）。
- **Crash-recovery**: 止まって戻る。
- **Omission**: メッセージを取りこぼす。
- **Byzantine**: 任意の振る舞い（悪意・故障）。ブロックチェーンで重要。

### FLP 不可能性

非同期 + 1 ノード故障で **決定的な合意は不可能**（Fischer, Lynch, Paterson 1985）。実用ではタイムアウトや乱択でこれを回避する。

## 16.4 一貫性モデル

「読み手はどんな値を見ることを許されるか」を規定。

- **Strict Consistency / Linearizability**: グローバル時計の即時反映。最強。
- **Sequential Consistency**: 順序は維持されるが時刻はずれてよい。
- **Causal Consistency**: 因果関係のあるイベントは順序を保つ。
- **Read-Your-Writes**: 自分の書いたものは読める。
- **Monotonic Reads**: 一度新しい値を見たら古い値を見ない。
- **Eventual Consistency**: 最終的に一致。

強い一貫性ほど可用性・性能を犠牲にする。

## 16.5 CAP と PACELC

**CAP**: ネットワーク分断 (P) のとき、Consistency と Availability を同時には満たせない。
**PACELC**: 分断時 (P) は C か A、それ以外 (E) は遅延 (L) と整合性 (C)。

- CP の例: HBase, Spanner, etcd。
- AP の例: Cassandra, DynamoDB (eventual モード)。

## 16.6 レプリケーション

データのコピーを複数ノードに置く。

### 同期 vs 非同期

- 同期: 全レプリカへ書込み完了で OK。安全だが遅い。
- 非同期: マスターのみ書込みで OK。速いが障害でデータロス。
- Quorum: $W + R > N$ で読書きが必ず最新を見る。

### マスター/スレーブ

- Single Leader: 書込みは 1 ノード。Postgres Replication, MySQL の半同期。
- Multi-Leader: 複数で書込み。地理分散・オフラインクライアントで使う。衝突解決が必要。
- Leaderless: Dynamo, Cassandra。$N, R, W$ で調整。

### 衝突解決

- Last-Write-Wins (LWW): 単純だがデータ損失。
- ベクトル時計 + アプリ層マージ。
- CRDT (Conflict-free Replicated Data Type): 自動的に収束する代数構造。Riak, Redis CRDB。

## 16.7 シャーディング (パーティショニング)

データを範囲・ハッシュで分割。

- レンジパーティション: 範囲クエリに強いが偏りやすい。
- ハッシュパーティション: 均等だが範囲クエリが苦手。
- 一貫性ハッシュ (Consistent Hashing): ノード追加時の再分配を最小化。
- 仮想ノード: 偏りをならす。

ホットスポット問題（人気キー）はソルトやキー分割で回避。

## 16.8 合意アルゴリズム

複数ノードが「同じ値に合意する」プロトコル。

### Paxos

Lamport が提案。Proposer / Acceptor / Learner。難解で有名。Multi-Paxos でリーダーを固定し効率化。

### Raft

Paxos の理解しやすい代替（Ongaro & Ousterhout 2014）。
- リーダー選出（タイマーランダム化）。
- ログレプリケーション（リーダーがフォロワに伝播）。
- 安全性（リーダー完備性、コミット制限）。

etcd, Consul, CockroachDB, TiKV が実装。

### Byzantine Fault Tolerance

PBFT, Tendermint, HotStuff。$3f + 1$ ノードで $f$ ビザンチン故障に耐える。ブロックチェーンでは PoW / PoS と組合せ。

### Gossip プロトコル

Epidemic 的に情報拡散。Cassandra のメンバーシップ、SWIM。

## 16.9 分散トランザクション

### 2 相コミット (2PC)

1. 準備フェーズ: コーディネータが全参加者に投票要求。
2. コミット/中止フェーズ: 全員 yes ならコミット。

ブロッキング（コーディネータ障害で参加者が永遠に待つ）。

### 3 相コミット

ブロッキングを軽減するが、ネットワーク分断には弱い。

### Paxos Commit

各シャードに Paxos を走らせコーディネータの単一障害点を解消。Spanner で採用。

### Saga パターン

ステップごとに「補償アクション」を定義し、失敗時に巻き戻す。マイクロサービスでよく使う。完全な ACID は保てないが、最終整合性は実現できる。

## 16.10 メッセージング保証

- **At-most-once**: 重複はないが消失あり。
- **At-least-once**: 必ず届くが重複あり。最も実装しやすい。
- **Exactly-once**: 1 度だけ。実は条件付き（消費者の冪等性必須）。

冪等な処理 + リトライで at-least-once を実用上 exactly-once 相当にできる。

### キュー / Pub-Sub

- Kafka: ログ指向、順序保証、リテンション、再生可能。
- RabbitMQ: AMQP、ルーティング豊富。
- SQS / SNS。
- NATS, Redis Streams。

### Outbox パターン

DB トランザクションとメッセージ送信を整合させる。DB に「送るメッセージ」を書き、別プロセスがキューに転送。

## 16.11 ストリーム処理

- 連続するイベントのリアルタイム集計。
- フレームワーク: Kafka Streams, Flink, Spark Streaming, Beam。
- ウィンドウ（タンブリング、スライディング、セッション）。
- イベント時間 vs 処理時間、ウォーターマーク。
- 厳密 1 度: チェックポイント + トランザクショナル書込み。

## 16.12 サービスメッシュとマイクロサービス

- ロードバランス、サーキットブレーカー、リトライ、タイムアウト、バルクヘッド。
- サービスディスカバリ (Consul, Kubernetes Service)。
- Istio, Linkerd: サイドカープロキシで横断的関心事をオフロード。
- gRPC, GraphQL, REST。

## 16.13 障害対策パターン

- リトライ + 指数バックオフ + ジッタ。
- サーキットブレーカー: 連続失敗で短絡し回復を待つ。
- バルクヘッド: 部分障害が他を巻き込まないよう隔離。
- レート制限・スロットリング。
- タイムアウト（必須、無限待ちはバグ）。
- 冗長化と自動フェイルオーバ。
- カオスエンジニアリング（Chaos Monkey）。

## 16.14 観測とデバッグ

- 分散トレーシング (OpenTelemetry, Jaeger, Zipkin)。
- 構造化ログ + トレース ID 伝播。
- メトリクス（リクエスト率、エラー率、レイテンシ p50/p99）。
- 因果関係追跡（Vector Clock の現代版）。

## 16.15 セキュリティ

- mTLS, ゼロトラスト。
- API ゲートウェイで集中認証。
- 機密管理 (Vault, KMS)。
- ネットワークポリシー (Calico, Istio AuthorizationPolicy)。

## 16.16 演習

1. ベクトル時計を 3 ノードでシミュレートし、独立イベントと因果イベントを区別せよ。
2. CAP の AP システムでネット分断時に「読みは古い値」を返す具体例を述べよ。
3. Raft でリーダー選出が起こる契機と、スプリットブレインを防ぐ仕組みを説明せよ。
4. 一貫性ハッシュが「ノード追加で 1/N のキーだけ移動」となる理由を述べよ。
5. 2PC のブロッキング障害シナリオを 1 つ示せ。
6. Saga で「注文 → 決済 → 配送」を構築し、決済失敗時の補償を書け。
7. At-least-once の上で exactly-once 相当を実現するために必要な「冪等性キー」の設計を提案せよ。
8. サーキットブレーカーが「半開」状態を持つ理由を説明せよ。

## 16.17 まとめ

分散システムは「障害は前提」「時間は信用するな」「すべては最終整合」というマインドセットの下で設計する。CAP・合意・レプリケーション・冪等性・観測性、これらが現代のクラウドネイティブの言語。**最初から分散にするな** が最も大事な格言: 単純なモノリスで足りる規模では、分散の複雑さは本末転倒な負債になる。必要になってから慎重に分散化する。
