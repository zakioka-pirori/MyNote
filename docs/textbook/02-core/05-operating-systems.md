# 第 10 章 オペレーティングシステム

## 学習目標

- OS が「ハードウェアの抽象化」と「資源の調停」という 2 つの仕事をしていることを理解する。
- プロセス・スレッド・スケジューリング・同期・デッドロック・メモリ管理・ファイルシステムを体系的に説明できる。
- システムコール、カーネル/ユーザ空間、コンテキストスイッチが何をしているかが分かる。
- 仮想化・コンテナ・並行プログラミングの基礎を獲得する。

## 10.1 なぜ OS を学ぶか

OS はあらゆるアプリの土台で、性能・安全性・保守性のすべてに影響する。「なぜ複数プロセスが同時に動けるのか」「メモリはなぜ無限に使えるように見えるのか」「ファイルとディレクトリの正体は何か」「`sleep(1)` で何が起きるのか」――これらすべて OS の話。低レイヤのバグを追えるエンジニアと、ブラックボックスで止まるエンジニアの差は、OS 知識の有無で決まる。

## 10.2 OS の役割

1. **抽象化**: CPU・メモリ・ディスク・ネットワークを使いやすいインターフェースで提供。
2. **資源管理**: 複数のプロセスに公平・効率的に資源を割当。
3. **隔離と保護**: プロセス同士、ユーザ同士の干渉を防ぐ。

カーネルとユーザ空間が **特権レベル** で分離される（CPU のリングまたはモード）。ユーザ空間からカーネルへ移るには **システムコール** を使う。

## 10.3 プロセスとスレッド

### プロセス

実行中のプログラム + その状態。各プロセスは:
- 独自のアドレス空間
- ファイルディスクリプタ
- 環境変数
- プロセス ID (PID)
- 親 PID

を持つ。Unix では `fork()` で複製、`exec()` で別プログラムに置き換え、`wait()` で子の終了を待つ。

### スレッド

プロセス内の実行単位。同じアドレス空間を共有する。

- ユーザレベルスレッド: ライブラリで管理（軽量だが OS から見えない）。
- カーネルレベルスレッド: OS が管理（POSIX threads, Windows threads）。
- M:N モデル: ユーザ M 本をカーネル N 本に多重化。Go の goroutine が近い。

### コンテキストスイッチ

CPU を別プロセス/スレッドに切り替える操作。レジスタ・PC・スタック・ページテーブル参照を保存・復元。コストが高い（数 µs）。

### プロセスの状態

```
        +---------+
+------>| Running |--------+ (timer / I/O)
|       +----+----+        v
| (resume)   | (exit)     +---------+
|            v            | Waiting |
|       +----------+      +----+----+
+----+--+ Ready    |<---------+ (I/O 完了)
     |  +----------+
     +-- (preempt)
```

## 10.4 スケジューリング

CPU を Ready キューのプロセスに割り当てる戦略。

### 評価指標

- スループット
- 応答時間
- ターンアラウンド時間
- 待機時間
- 公平性

### 古典アルゴリズム

- **FCFS** (First-Come First-Served): 単純、コンボイ効果。
- **SJF** (Shortest Job First): 待機時間最小化、ただし長いジョブが飢餓。
- **ラウンドロビン**: タイムスライスで公平。
- **優先度**: 数値で順位付け。
- **MLFQ** (Multi-Level Feedback Queue): 動的に優先度を上下。Linux の CFS、Windows の優先度ベースなど現代 OS の基盤。

### Linux CFS (Completely Fair Scheduler)

赤黒木で「実行時間が最少」のタスクを次に実行。仮想実行時間 (vruntime) を nice 値で重み付け。

### リアルタイム

- ハードリアルタイム: 締切違反は致命的（医療、航空）。
- ソフトリアルタイム: 締切違反は望ましくないが許容（動画再生）。
- EDF (Earliest Deadline First)、Rate Monotonic。

## 10.5 同期

並行性は **競合状態 (race condition)** を生む。

### クリティカルセクション

1 度に 1 スレッドだけが入れる領域。**相互排他 (mutual exclusion)** を保証する原始操作:

- TestAndSet, CompareAndSwap (CAS): ハードウェアの不可分命令。
- ミューテックス (lock/unlock)。
- セマフォ (P/V 操作、計数)。
- モニタ (mutex + condition variable)。
- 読み書きロック。
- スピンロック vs ブロッキングロック。

### 条件変数

「ある条件が成り立つまで待つ」仕組み。`wait`, `signal`, `broadcast`。生産者・消費者問題、有限バッファで使う。

### バリア・セマフォの応用

- 食事する哲学者問題（5 本のフォークと 5 人の哲学者、デッドロック例）。
- 読者・書き手問題。
- 寝ている理髪師問題。

### メモリモデル

CPU はメモリ操作を再順序化する。複数スレッドで共有変数を扱うときは:
- メモリバリア（fence）
- アトミック変数（C++ の `std::atomic`、Java の `volatile`）
- happens-before 関係

を意識する必要がある。

### ロックフリー / Wait-free

CAS だけで構築する高度な同時並行データ構造。複雑だが性能・耐障害性で勝る。

## 10.6 デッドロック

4 つの必要条件（Coffman 条件）:
1. 相互排他
2. 保持と待ち
3. 横取り不可
4. 循環待ち

### 対処戦略

- **予防**: 4 条件のいずれかを破る（資源を順序付けて要求する等）。
- **回避**: バンカーアルゴリズム。資源要求を安全状態が保たれる場合のみ許可。
- **検出と回復**: 待機グラフでサイクル検出、プロセスを kill。
- **無視（ダチョウ算法）**: 実装が複雑なので発生時に再起動。デスクトップ OS の現実解。

## 10.7 メモリ管理

### 連続割当（古典）

- 単一プログラム時代の方式。
- 断片化（外部・内部）。

### ページング

仮想アドレス空間を固定サイズのページに分割し、物理フレームに対応付ける。
- ページテーブル: 仮想ページ番号 → 物理フレーム番号 + 属性（R/W/X、有効ビット、ダーティ）。
- 多段ページテーブル: x86-64 は 4 段。
- TLB: ページ翻訳のキャッシュ。

### ページ置換

物理メモリ満杯のときに「どのページを追い出すか」:
- FIFO, LRU, LFU, Clock, ARC。
- 最適アルゴリズム（OPT, Belady）は理論上界、実装不能（未来を知る必要がある）。

### スラッシング

ワーキングセットが物理メモリに収まらず、ページフォルトばかりが起こる状態。プロセスを減らすか物理メモリを増やすしかない。

### メモリ割当（malloc 内部）

- フリーリスト、ベストフィット、ファーストフィット。
- スラブアロケータ（カーネル内）。
- jemalloc, tcmalloc。
- ガベージコレクション（マーク&スイープ、世代別、参照カウント、コンカレント GC）は第 13 章で。

## 10.8 ファイルシステム

### 抽象

- ファイル（バイトの並び + メタデータ）。
- ディレクトリ（名前 → ファイルの対応）。
- リンク（ハード・シンボリック）。
- パス、マウント。

### 実装

- スーパーブロック、inode、データブロック。
- 直接ブロック、間接ブロック、二重間接、三重間接。
- ジャーナリング (ext4, XFS): 書き込みをログに先に書いて整合性を保証。
- COW (Copy-On-Write): ZFS, Btrfs, APFS。スナップショットが安価。
- 分散ファイルシステム: NFS, GFS, HDFS, Ceph。

### POSIX API

`open`, `read`, `write`, `lseek`, `close`, `stat`, `unlink`, `mkdir`, `mmap`。

`mmap` でファイルを仮想アドレス空間にマップすると、read/write を介さず直接アクセスできる（DBMS や巨大データの読み込みで重要）。

## 10.9 入出力

### 同期 / 非同期

- 同期 I/O: 完了まで呼び出しスレッドが待つ。
- 非同期 I/O: 完了通知で続きを実行。Linux: `io_uring`、Windows: IOCP、BSD: `kqueue`。

### イベント駆動

`select`, `poll`, `epoll`, `kqueue` で多数のファイルディスクリプタを 1 スレッドで扱う。Node.js, Nginx, Redis の高並列性の正体。

### バッファリング

ユーザバッファ、カーネルバッファ、ディスクキャッシュ（ページキャッシュ）。

## 10.10 仮想化とコンテナ

### 仮想マシン (VM)

ハイパーバイザがハードウェアをエミュレート（KVM, Xen, VMware）。
- タイプ 1: ベアメタル。
- タイプ 2: ホスト OS の上で動く。
- ハードウェア支援 (Intel VT-x, AMD-V)。

### コンテナ

OS カーネルを共有し、ファイルシステム・ネットワーク・PID 空間だけ隔離。
- Linux の名前空間 (namespace) と cgroup。
- Docker, Podman, containerd。
- 軽量だが kernel を共有するため、互換性とセキュリティの境界が VM より弱い。

## 10.11 ブートとカーネル構造

- BIOS / UEFI → ブートローダ → カーネル → init。
- モノリシックカーネル: Linux, BSD。すべてカーネル空間。
- マイクロカーネル: Mach, L4。最小限のカーネル + ユーザ空間サービス。
- ハイブリッド: Windows NT、macOS。

## 10.12 セキュリティと隔離

- ユーザ・グループ・パーミッション (rwx)。
- 能力 (capability)、ACL。
- SELinux, AppArmor。
- システムコール制限 (seccomp)。
- ASLR, DEP/NX、スタックカナリア。

詳細は第 15 章で。

## 10.13 並行プログラミングの実践

- POSIX スレッド (pthread): `pthread_create`, `mutex`, `cond`。
- Java の `synchronized`、`Lock`、`Concurrent` パッケージ。
- Go の goroutine + channel。
- Erlang のアクター。
- Rust の所有権 + Send/Sync。

「**共有メモリで並行は危険。メッセージで並行は安全**」というアクター/CSP モデルの哲学を覚えておくとよい。

## 10.14 演習

1. `fork()` 後の親と子の挙動を、PID と変数の値で説明せよ。
2. ラウンドロビン (タイムスライス 4) で次のジョブ群の実行順序を求めよ: A=10, B=4, C=8。
3. 食事する哲学者問題で、フォークの取得順を統一すればデッドロックを避けられる理由を説明せよ。
4. ページサイズ 4 KB、仮想アドレス 32 ビット、物理メモリ 1 GB のとき、単一段ページテーブルのサイズを計算せよ。
5. LRU と Clock の動作を簡単な参照列で比較せよ。
6. `strace ls` を実行し、出力されるシステムコールを 10 個分類せよ。
7. Linux の `epoll` と `select` の計算量を比較し、なぜ高並列で `epoll` が選ばれるか説明せよ。
8. Docker コンテナと VM のセキュリティ境界の違いを述べよ。

## 10.15 まとめ

OS は「ハードウェアを使いやすく見せ、資源を巧妙に分配する」中間層。プロセスとスレッド、スケジューリング、同期、メモリ管理、ファイルシステム、I/O モデル、仮想化――この 7 つを腹に落とすと、ありとあらゆる性能問題・並行性バグ・セキュリティ問題に立ち向かえる。**`man 2 syscall` をすぐ引ける** 状態を目指そう。
